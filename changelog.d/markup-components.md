feature: **Markup components — cross-file imports, arg defaults, and slots**: `<import src="components/board-column.zml"/>` at the top of a `.zml` file splices a component file's templates (transitively, in import order, exactly as if pasted at the import site) before your own — paths resolve relative to the importing file under the markup root, cycles report the cycle path, duplicate template names name both definition sites, and an imported file defines templates only (a template-only component file is valid standalone). Template args now take literal defaults (`args="title trend=flat"`; use sites may omit them, `{binding}` defaults are teaching errors), and a template body can mark one `<slot/>` where use-site children are inserted — slot content builds in the CONSUMER's scope (it sees the use site's model paths and loop variables) and structural widget ids hash exactly as if the content were written inline.
- **Both engines, one resolver seam**: the compiled engine takes an embedded source set via `canvas.CompiledMarkupImports(Model, Msg, "root.zml", &sources)`, the runtime interpreter resolves the same set for embedded sources (`MarkupOptions.sources`) and the file system for hot reload — so `native dev` reloads on edits to IMPORTED files too — and parity tests pin identical trees, ids, and dispatch across imports, defaults, and slot content (including a `<use>` with a defaulted arg inside slot content).
- **`native markup check` follows imports**: checking a view checks its whole import closure, a broken import reports at the importing file's `path:line:column`, component files check standalone, and hostile inputs (self-import, deep chains, huge template counts) get bounded teaching errors. The kanban example ships the adoption: `board-column` extracted to `src/components/board-column.zml` with pixel-identical widget ids.
