fix: **Linux audio buffering flag no longer sticks across a no-op resume**: the flag normally drops when the pipeline's PLAYING state-changed message arrives, but when a mid-track refill resolved before its own pause request ever completed, the 100%-buffered resume was a no-op that posted no transition message — and "buffering…" rode every position tick for the rest of the track while audio played at full speed. The resume now reads the state-change result the way the backend defines it: a non-async return means the pipeline is at the target state already and no message is coming, so the flag drops immediately; an async return leaves the drop to the state-changed handler, emitted the moment PLAYING is actually reached.
