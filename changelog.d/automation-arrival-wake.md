fix: **Automation commands wake an idle app instead of waiting for a frame**: the runtime drains one dropbox command per `frame_requested` turn, but a quiet app (no animation, no input, no timer about to fire) produced no frames — so a queued `native automate widget-click` could sit past the CLI's 10 s consumption wait. A command LANDING now wakes the loop the way user input does: with automation on, a dedicated arrival watcher polls the command slot off the loop thread and asks the platform for one coalesced frame tick through a new thread-safe `request_frame_fn` service (macOS: main-queue dispatch — deliberately timer-free, so it works even while the OS coalesces a backgrounded app's timers; GTK: `g_idle_add`; Win32: `PostMessage`). Consumption stays on the frame boundary, so command order, one-command-per-frame, and session record/replay shape are unchanged — measured on the idle gpu-components example, widget-click consumption went from ~40-50 ms (the old 20 Hz automation poll timer, now removed) to ~4 ms, and builds without `-Dautomation=true` carry no thread, no polls, and no wakes.
- **Undispatchable command lines free the slot**: a malformed or oversized line in `command.txt` is now acked to `done` and recorded as a dispatch error the snapshot surfaces, instead of silently stranding the single-entry slot (which blocked every later command and its CLI wait).
