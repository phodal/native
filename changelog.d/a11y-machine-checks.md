feature: **Accessibility as machine checks**: "every interactive element is properly labeled" is now enforced, not reviewed. The markup lint (in `validate`, so `native markup check`, both view engines, and the LSP all report it) makes an unnamed interactive control, an icon-only control without a `label` (the icon name is a drawing instruction, not a label), an unknown literal `role`, or a container role on an element that can never hold children (`role="tree"` on a text leaf) a validation error — the rubric is whether a screen reader user is fully blocked; unnamed images/avatars and labels duplicating the text they shadow degrade rather than block, so they report as warnings (`--strict` promotes them, `label=""` marks an image decorative). Element classes are registry data (`ui_schema.A11yNameRule`, the role vocabulary, container roles), conformance-tested against the engine's control predicates and the live `WidgetRole` enum.
- **Tree-level a11y audit** (`canvas.a11y`, `canvas.expectA11yAuditSweepClean`): a deterministic pass over the built widget tree — the layout audit's shape — catching what markup cannot see: builder-authored views and dynamic labels that resolve empty at runtime (`missing_label`, judged by the role and label the bridges actually announce), focusable widgets keyboard traversal can never reach because a non-scrolling clip ancestor clipped them out (`focus_unreachable`), and identically labeled sibling controls a screen reader cannot tell apart (`duplicate_sibling_label`). Adopted by every showcase suite next to the layout sweep; the sweep found and fixed a real ambiguity (feed timeline rows labeled only by author, identical when an author posts twice).
- The LSP publishes the warning half at LSP warning severity once a document validates clean, and `native markup check` prints warnings with positions per file.
