fix: **Linux GPU surfaces no longer freeze under a saturated frame loop**: the per-surface frame-event emission now schedules at idle priority, below GTK's layout and paint sources, instead of plain `g_timeout_add`'s default priority. When an armed animation's cycle cost exceeded one frame interval (a large software-rendered canvas), every present re-armed the emission at zero delay and the always-ready default-priority timeout starved layout and paint entirely — presents kept landing in the retained buffer and input kept dispatching, but the visible window froze on stale pixels for minutes (a mid-stall resize did not even re-layout). At idle priority the pending paint and relayout win each cycle, so every presented frame reaches the glass before the next frame event; the demand-driven idle law (an idle surface emits zero frame events) is unchanged.
