feature: **`native eject component <name>`**: transfer ownership of a library composite to your app. The default stays use-and-theme; ejecting writes the composite's canonical source into `src/components/` — exactly once, never overwriting — for the moment you need to own its shape. The v1 set is the honest compositions: `stepper` and `timeline-item` eject as Zig view functions (their conditional structure and formatted text sit outside the closed markup grammar), `timeline` ejects as a markup `<template>` reached through `<use template="timeline" ...>`, so it never shadows the built-in element. Engine controls are not on the menu — their behavior lives in the runtime, and the way to change them is the token system, not a fork.
- **Held honest by widget-identity tests**: each canonical source builds against the same inputs as its library form and must produce an identical widget tree — every id, field, and handler — so ejecting is an ownership change, never a visual one (`zig build test-eject-components`).
- **Teaching errors**: ejecting twice says `already ejected at src/components/<file> - delete it to re-eject`; unknown names get a did-you-mean plus the ejectable list; running outside an app directory errors like the other verbs.
- Every ejected file opens with a header comment that walks through migrating call sites and notes that deleting the file falls back to the library form.
