feature: **Present-before-show for canvas windows**: a window whose content is a canvas (any `gpu_surface` view — the startup window, scene windows, and model-declared `windows_fn` windows alike) is now created ORDERED OUT and becomes visible only after its first canvas frame has completed presentation, so opening a window never flashes blank — the blank-settings-window launch gap is gone by construction. The runtime derives the policy from the window's views (`WindowOptions.show = .on_first_present`; webview windows keep immediate show — their engine owns first paint), the macOS host orders the window front from the first successful gpu-surface present (measured live: startup window 96 ms create→shown-with-content, settings window 203 ms first open / 54 ms reopen — intervals that were previously all visible blank), a 1 s fallback deadline guarantees a wedged first frame surfaces as a late window rather than an invisible app, and `NATIVE_SDK_WINDOW_TIMING=1` logs each window's create→show latency.
- **Canvas windows wear the app's background**: every packet present now sets `NSWindow.backgroundColor` from the packet's clear color (the runtime already records it per view), so any residual gap — resize slack, the titlebar band before content lands — shows the app's background instead of the system default. The null platform records per-window show policy, live visibility, and present/shown sequence numbers, and the ordering contract (first present strictly before visibility) is regression-tested through the real create seam.
