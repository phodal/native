feature: **Titlebar control lands end to end**: `titlebar = "hidden_inset"` now reaches the STARTUP window — app.zon's `.windows[0]`/`.shell.windows[0]` declaration (validated by `zig build validate`) threads through `native_sdk_appkit_create`, which previously hardcoded standard chrome for window 1 — and the app's own header honestly replaces what the titlebar did. A `window-drag="true"` element (Zig: `.window_drag`) is a window-drag surface in both markup engines: pressing its background or plain text inside hands the pointer-down to `performWindowDragWithEvent:` (moves only on actual movement; a plain click is a no-op), press fall-through keeps buttons inside fully clickable, and double-click applies the user's titlebar double-click preference (zoom by default). The new `on_chrome` UiApp hook delivers the chrome overlay insets (titlebar band height, traffic-light extent — live AppKit geometry, so fullscreen honestly reports zero) before the first view build and on changes, so headers pad their leading edge instead of hardcoding pixel counts. The CEF host carries the same symbols in sync (including the `titlebar_style` parameter `native_sdk_appkit_create_window` had drifted without); GTK/Win32 are scoped out deliberately like resizability — standard chrome, dead-space drags, zero insets. The null platform records per-window titlebar styles and every `startWindowDrag` call, and serves settable chrome insets, so all three channels are regression-tested through real dispatch. `examples/markdown-viewer` ships the retrofit: hidden-inset chrome, its toolbar as the drag region, traffic-light padding from `on_chrome`.
