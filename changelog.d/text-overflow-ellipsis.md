feature: **Single-line text elides with a trailing ellipsis by default** — a plain `text` leaf or a control label (button, badge, select value, tooltip, menu/list rows, table cells, tabs, status bar, accordion header) whose content does not fit its box now paints the widest prefix that fits plus a trailing … taken from the run's own face (U+2026; a registered face lacking it takes the documented notdef fallback), instead of clipping mid-glyph or re-wrapping over the row below. The elision point is computed on the same measurement seam paint uses (the host provider when installed, the deterministic estimator otherwise), so the painted extent never exceeds the frame; the un-elided `text` bytes stay the selection/copy source of truth (an elided line still maps every hidden byte — a rightward sweep selects them, copy gets the full string), and packet hosts draw the elided lines verbatim (kept prefix + marker), no host-side elision logic. `overflow` is the new policy knob (markup `overflow=` on `text`, `ElementOptions.overflow`, `TextLayoutOptions.overflow`): `ellipsis` is the default; `clip` keeps the honest hard-cut for fixed-format content (a duration column, where "1…" would be worse than a partial glyph); overflow-visible is deliberately not offered — painting past the frame is the bug class the layout audit exists to catch. `wrap="true"` paragraphs keep wrapping (no mid-paragraph ellipsis); `wrap="false"` and unset are both the honest single line, now eliding by default. Text inputs never elide (their overflow scrolls into reach). The layout audit updates in kind: an ellipsized line is correct rendering, never a finding — what reports is suppressed elision (`overflow="clip"`) that actually truncates, and span paragraphs squeezed below their reserved wrapped extent. Both markup engines, the validator vocabulary (with teaching errors), contracts, the LSP attribute docs, the skill, and the docs cover the new attribute.
