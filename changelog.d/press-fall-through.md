feature: **Presses fall through to the nearest pressable ancestor — the container-press trap is dead**: hit-testing still resolves the deepest widget under the pointer (hover, cursor, and text selection need it), but press dispatch now walks the hit path to the deepest widget that CLAIMS presses — an interactive kind, or ANY widget with a bound `on_press`/`on_toggle` (the handler itself makes an element a widget-level hit target, so `ui.row(.{ .on_press = ... })` and `<row on-press=...>`/`<panel on-press=...>` are real pressable rows now). Plain text, icons, images, badges, and layout containers let clicks fall through; nested pressables resolve to the deepest one (a button inside a pressable row wins); editable text fields, scroll containers, and dismissible surfaces (dialog, drawer, sheet, popover, menus) always claim their own presses. Text selection keeps working inside pressable rows via press-vs-drag disambiguation: dragging selects (and presses nothing), a plain click collapses the selection and lands on the row. The resolution happens at pointer routing (`CanvasWidgetPointerEvent.press_target`), so typed `on_press` dispatch, engine-owned `command` dispatch, control activation (list-item selection), focus-on-press, and automation `widget-click` all agree with the live hit test by construction. The empty-text-overlay/duplicated-handler workaround idiom is retired everywhere it lived: `ui.timelineItem` binds its press to the item root (no overlay child), the notes example's folder and note rows are plain pressable panels, and the markup validator/engines/LSP now accept `on-press`/`on-toggle` on any element while still rejecting the genuinely dead `on-change`/`on-submit`/`on-input` on layout/decoration elements with an updated teaching message.
