fix: **Blank main window on oversized inline sources, loudly diagnosed**: the webview example's main window came up blank under both web engines because its inline HTML had outgrown the runtime's window-source copy budget — `loadStartupWindows` failed with `WindowSourceTooLarge` during `app_start`, and the failure was invisible: the error unwound into the platform stop path, the trace read like a clean `app.stop`, and since the stop request landed before the run loop existed the app kept running behind a blank window. Four repairs: the inline-source budget is now 64 KiB (`max_window_source_bytes`, with the path-shaped asset fields split onto their own 4 KiB `max_window_source_path_bytes` so per-window storage stays lean), a startup failure now records a `dispatch.error` trace naming the failing event and error, both macOS hosts honor a pre-run-loop shutdown request instead of stranding a live blank window, and the webview smoke suites (both engines) now assert the main window actually loaded its source (`webview.load` present, no `dispatch.error`) instead of only exercising the automation bridge channel — plus an example unit test pins the inline HTML inside the budget.
