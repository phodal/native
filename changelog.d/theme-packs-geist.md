feature: **Theme packs + the register-token tier**. The last code-owned look decisions moved into the token system, with the current behavior as byte-identical defaults (the pinned catalog and dashboard reference signatures did not move): interaction-state formulas live in the new `states` tokens (filled hover/pressed alpha cuts, the disabled wash strength, the static-text selection wash, the destructive chip's wash ladder), the control size register lives in the new `metrics` tokens (the sm/default/lg height ladder, per-rung button insets, button label steps, icon extents/gaps, row extent), focus ring geometry is fully token-stated (`stroke.focus` width + the new `stroke.focus_offset` gap), and the per-control tables grew `pressed_background`, `disabled_background`, and `disabled_foreground` channels — so a design system whose press feedback or disabled treatment is a color swap states it as data, never an emitter patch. On top of that tier, named theme packs: a pack is a complete `DesignTokens` register (light, dark, and both high-contrast variants) selectable by `ThemeOptions.pack`, an app.zon `theme` field (unknown names are build/check errors), or the `UiApp` `theme` option — packs compose with the live system appearance, so a packed app still re-themes on the OS light/dark flip. The first built-in pack beyond the house register is `geist` (the design register of the bundled Geist typeface family: cool neutrals over pure white/black, monochrome primaries, blue focus rings, 6px control corners, a 32/40/48 control ladder), and the component catalog is now pinned under BOTH packs — each design system the SDK ships is machine-verified pixel-for-pixel in CI, the same pattern apps can use to pin their own.
